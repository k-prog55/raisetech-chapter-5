# HTTPにまつわる用語

## URLとは？　
ホームページの「**住所**」に相当するものであり、ホームページが置いてある場所やファイル名を示す情報のこと。WebブラウザからWebサーバに対して情報のアクセスをする際にアクセス場所を示す役割を果たしている。
` 「Uniform Resource Locator（ユニフォーム・リソース・ロケータ）」の略。`

## クエリ文字列とは？
クエリ文字列は、URL内のデータを伝えるための文字列。「？」で始まり、キーと値のペアは「=」で結ばれ、複数のペアは「&」で区切られる。
例：https://example.com/search?q=apple&category=fruits

## パスパラメーター（パス変数）とは？
パスパラメーターは、URLの可変データを表す仕組み。例：https://example.com/users/12345の"12345"がパスパラメーター。サーバーが動的にデータを処理・表示。便利でシンプルな方法。

## パスパラメーターとクエリ文字列との違いは？
パスパラメーターとクエリ文字列は、ウェブアドレスでデータを伝える方法を示す。

パスパラメーター：URLのパス部分に含まれるデータ。例えば、「https://example.com/users/12345」の「12345」がパスパラメーター。サーバーはデータを処理し、対応するページを表示する。URLがシンプルで直感的。

クエリ文字列：URLの末尾に「？」を使い、キーと値のペアを「=」で結び、「&」で区切る。例えば、「https://example.com/search?q=apple&category=fruits」の「q=apple」や「category=fruits」がクエリ文字列。ユーザーがデータをサーバーに送り、結果を取得する。検索やフィルタリングに便利。

パスパラメーターはURLの一部で、直感的なリンクを作るのに適している。クエリ文字列はデータの送受信に適しており、特定の操作を実行したい場合に使われる。

## HTTPメソッドとは何か
GET/POST/PUT/PATCH/DELETEそれぞれの意味を調べましょう

- GETメソッド：
事例：ブラウザで「https://example.com/products/12345」にアクセスするときにGETメソッドが使用されます。
用途：ウェブサーバーから商品IDが「12345」の商品の情報を取得します。

- POSTメソッド：
事例：ユーザーがフォームに情報を入力して、「https://example.com/users」に送信するときにPOSTメソッドが使用されます。
用途：ウェブサーバーに新しいユーザーデータを送信して、ユーザーアカウントを作成します。

- PUTメソッド：
事例：APIを使用して「https://example.com/products/12345」に対してPUTメソッドを使用すると、商品IDが「12345」の商品情報が更新されます。
用途：指定したURLのリソース（この場合は商品情報）を完全に置き換えます。

- PATCHメソッド：
事例：SNSの投稿に対して「https://example.com/posts/56789」にPATCHメソッドを使用すると、投稿IDが「56789」の投稿の一部を更新します。
用途：指定したURLのリソースの一部を変更・更新します。

- DELETEメソッド：
事例：オンラインショッピングサイトで「https://example.com/cart/items/9876」に対してDELETEメソッドを使用すると、カート内の商品IDが「9876」の商品を削除します。
用途：指定したURLのリソース（この場合はカート内の商品）を削除します。

## HTTPステータスコードとは何か
下記のHTTPステータスコードの意味を調べましょう。

- 200 OK（成功）:
これはリクエストが成功したことを示します。例えば、ウェブページの読み込みが成功したり、フォームの送信が成功した場合に使われます。

- 201 Created（作成）:
新しいデータやリソースがサーバー上に作成されたことを意味します。例えば、新しいアカウントが作成されたり、新しい投稿が作成された場合に返されます。

- 400 Bad Request（リクエストエラー）:
リクエストがサーバーによって理解できなかったり、問題があったりする場合に返されるエラーコードです。例えば、URLが間違っている、必要な情報が不足している、またはリクエストの形式が正しくない場合に使われます。

- 404 Not Found（ページが見つかりません）:
リクエストされたページやリソースがサーバー上に見つからなかった場合に返されます。URLが間違っているか、ページが削除された可能性があります。

- 500 Internal Server Error（サーバーエラー）:
サーバーで何らかの問題が発生して、リクエストを処理できなかった場合に返されます。具体的なエラーの原因が不明な場合や、サーバーがうまく動作していない場合に表示されます。

## リクエストヘッダーとは何か
リクエストヘッダー（Request Header）は、クライアントがウェブサーバーに対してリクエストを送信する際に、リクエストに関するさまざまな情報や設定を含む部分です。HTTP（Hypertext Transfer Protocol）通信において、リクエストヘッダーはリクエストの性質やクライアントの要求をサーバーに伝えるために使用されます。

リクエストヘッダーには以下のような情報が含まれます：

- メソッド（Method）: リクエストのタイプを示すもので、一般的なものとしてはGET、POST、PUT、DELETEなどがあります。クライアントがサーバーに対して何を行いたいかを指定します。
- ユーザーエージェント（User-Agent）: クライアントが何のアプリケーションやデバイスを使ってリクエストを送信しているかを示す情報です。ウェブブラウザ名やバージョンが含まれることがあります。
- コンテンツタイプ（Content-Type）: リクエストの本文（リクエストボディ）の形式を指定します。主にPOSTやPUTメソッドで使われます。
- ホスト（Host）: クライアントがアクセスしたいウェブサーバーのホスト名やIPアドレスを指定します。特に1つのサーバーが複数のウェブサイトを提供する場合に必要です。
- クッキー（Cookie）: クライアントがサーバーに保存されているクッキー情報を含めることができます。サーバーはこれを使ってクライアントを識別したり、セッションを管理したりすることができます。

他にも、リクエストヘッダーには認証情報や圧縮方式の指定、キャッシュの制御など、さまざまな情報が含まれることがあります。これらの情報はクライアントとサーバーの間の通信を効率的に行い、適切な応答を返すために重要です。

## リクエストボディとは何か
リクエストボディ（Request Body）は、クライアントがウェブサーバーに対してデータや情報を送信する際に、HTTPリクエスト内に含まれる部分です。HTTP（Hypertext Transfer Protocol）通信において、リクエストボディはリクエストヘッダーの後に続き、サーバーに送りたい具体的なデータを含んでいます。

リクエストボディは主に以下のようなシナリオで使用されます：

- フォームデータの送信: ウェブサイトのフォームに入力した情報をサーバーに送信する場合に使用されます。例えば、ユーザーがウェブフォームに名前、メールアドレス、パスワードなどを入力し、「送信」ボタンを押すと、これらの情報がリクエストボディに含まれてサーバーに送られます。
- JSONデータの送信: クライアントがサーバーにJSON形式のデータを送信する場合にもリクエストボディが利用されます。これは、データの構造化された形式で情報を送信する際によく使われます。
- ファイルのアップロード: クライアントがウェブサーバーにファイルをアップロードする場合には、ファイル自体がリクエストボディに含まれます。これは、画像、動画、ドキュメントなどのファイルをサーバーに送信するために使われます。

リクエストボディの形式は、リクエストヘッダー内の「Content-Type」という項目で指定されます。よく使われる形式には、以下のようなものがあります：

- application/x-www-form-urlencoded: 通常、HTMLフォームで使用されるデフォルトのエンコーディング形式です。
- application/json: JSON形式のデータを送信する場合に使われます。
- multipart/form-data: ファイルのアップロードなど、複数のデータを送信する場合に使われる形式です。

リクエストボディは、クライアントがサーバーに対して必要な情報を提供するために重要な部分であり、ウェブアプリケーションの機能を実現するために利用されます。

## レスポンスヘッダーとは何か
レスポンスヘッダー（Response Header）は、ウェブサーバーがクライアントに対してリクエストに対する応答として送信する際に、レスポンスに関する情報を含む部分です。HTTP（Hypertext Transfer Protocol）通信において、レスポンスヘッダーはサーバーがクライアントに対して提供するさまざまな情報を伝えるために使用されます。

レスポンスヘッダーには以下のような情報が含まれます：

- ステータスコード（Status Code）: サーバーがリクエストを処理した結果を示す3桁の数字です。例えば、成功を示す200やリソースが見つからないことを示す404などがあります。
- コンテンツタイプ（Content-Type）: レスポンスの本文（レスポンスボディ）の形式を指定します。主にテキスト、HTML、JSON、画像、PDFなどの形式があります。
- クッキー（Cookie）: サーバーがクライアントに対して設定したクッキー情報が含まれることがあります。クライアントはこれを保存して次回のリクエストに含めることができます。
- キャッシュコントロール（Cache-Control）: クライアントやプロキシサーバーがレスポンスをキャッシュする方法を指定します。
- レスポンスの長さ（Content-Length）: レスポンスの本文のサイズ（バイト数）を示します。

他にも、レスポンスヘッダーにはセキュリティ関連の情報、サーバーの情報、リダイレクトに関する情報などが含まれることがあります。

レスポンスヘッダーは、クライアントがサーバーからの応答を正しく理解し、適切に処理するために重要な情報を提供します。例えば、ステータスコードはリクエストが成功したのか失敗したのかを示し、コンテンツタイプはレスポンスの内容を解釈する際に役立ちます。

## レスポンスボディとは何か
レスポンスボディ（Response Body）は、ウェブサーバーがクライアントに対してリクエストに対する応答として送信する際に、実際のデータやコンテンツが含まれる部分です。HTTP（Hypertext Transfer Protocol）通信において、レスポンスボディはサーバーがクライアントに提供する実際の情報やコンテンツを含むため、ウェブページの表示、画像のダウンロード、APIからのデータの取得など、さまざまな用途で使われます。

具体的な例としては、以下のようなものがあります：

- HTMLコンテンツ: ウェブサイトのページの内容がHTML形式でレスポンスボディに含まれます。ウェブブラウザはこのHTMLコンテンツを解釈してウェブページを表示します。
- JSONデータ: ウェブアプリケーションやモバイルアプリケーションがAPIを使ってデータを取得する際に、JSON形式でデータがレスポンスボディに含まれます。アプリケーションはこれを解析してデータを利用します。
- 画像ファイル: ウェブページ上で表示される画像は、画像ファイルとしてレスポンスボディに含まれます。ウェブブラウザはこれを取得して画像を表示します。
- テキストコンテンツ: ウェブサーバーがテキストファイルを提供する場合、その内容がレスポンスボディに含まれます。例えば、READMEファイルやプレーンテキストのドキュメントなどが含まれます。

レスポンスボディに含まれるデータは、サーバーがクライアントに提供したい情報によって異なります。クライアント（ウェブブラウザやアプリケーション）は、受け取ったレスポンスボディを適切に解釈して表示したり、データを処理したりすることで、ユーザーに意味のある情報や機能を提供します。

## JSONとは
JSON（JavaScript Object Notation）は、データを表現するための軽量で人間にも分かりやすい形式のデータフォーマットです。主にウェブアプリケーションやモバイルアプリケーションなどでデータを交換する際によく使われます。

JSONは、名前と値のペアを使ってデータを表現します。名前と値のペアはコロン（:）で区切り、各ペアはコンマ（,）で区切ります。

JSONの特徴としては以下が挙げられます：

- 簡潔で読みやすい: JSONは人間にとっても理解しやすい構造を持っています。
- データの表現: 配列（リスト）やオブジェクト（辞書）など、様々なデータ構造をサポートしています。
- プログラム言語に依存しない: JSONはプログラム言語に依存せず、多くのプログラム言語でパース（解析）や生成（生成）がサポートされています。
- ウェブAPIでよく使われる: ウェブアプリケーションの間でデータをやり取りする際に、JSONがよく使われます。特に、APIを使ってデータを取得したり送信したりする場面でよく利用されます。

JSONの柔軟な表現形式と広いサポートにより、データの交換や処理が簡単かつ効率的に行えるため、現代のウェブ開発において欠かせないデータフォーマットとなっています。
